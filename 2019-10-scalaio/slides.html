## Communication ++++

Communication:

 ## Good old rest
  • HTTP
  • Proper method
  • /user/{id}/cats
  • json
  • Very well known
  • Established standard
  • Not so efficient
  • Not so convenient for frontend
++++

Communication:

 ## Graphql * convenient for the fronted * not so much for the backend * nice code generation * a lot of tooling * live subscriptions OOTB ++++

Communication: Graphql

 ## How does it work? * Servers exposes schema (types, operations) * Client codebase contains queries * Code can be generated based on queries and schema ++++

Communication: Graphql

 ## In Scala? * Sangria - Seasoned, future-based * Caliban - New and shiny, ZIO-based, published for scala.js(!), server-only for now I recommend Caliban, because, you know, FP! ++++

Communication: Graphql

 ### Or maybe no backend at all? With tools like Prisma, PostGraphile and Hasura you can expose big part of the
API directly from the DB. ++++

Communication:

 ## RPC Remote Procedure Call
Features:
  • easily shared between fronted and backend
  • convenient for the backend
  • problematic on the fronted
  • usually requires single language
 Examples:
  • grpc-web
  • autowire
  • udash
++++

Communication:

 ## Shared REST
Features:
  • easily shared between fronted and backend if both are written in Scala
  • requires the same stack on both sides
  • all the benefits and problems of REST
 Examples:
  • endpoints
  • tapir
  • typedapi
++++

Communication:

 ## Websockets & SSE
Websockets:
  • on top of raw TCP
  • distinct from HTTP
  • bidirectional
  • binary
 Server-sent events:
  • raw HTTP
  • part of HTML5
  • one direction
  • text
  • with browser limits
Most of the time SSE is enough ----
